# Data Directory Directory (`data/`)

This directory serves as the local data persistence layer for the RAG chatbot system. It stores the preprocessed text cache, vector and lexical search indices, and the SQLite query monitoring logs.

---

## Files and Folder Structure

```
data/
├── chroma_db/              # Persistent ChromaDB vector store
├── bm25_index.pkl          # Serialized BM25 index and chunk mapping
├── corpus.db               # SQLite database caching document texts, pages, and chunks
├── monitoring.db           # SQLite database logging chatbot usage metrics
└── README.md               # This file
```

---

## Description of Components

### 1. SQLite Relational Cache (`data/corpus.db`)
* **What it is**: A SQLite database containing the extracted page-by-page text, chunk metadata, and document hashes from the 11 PDF documents in the `Docs/` directory.
* **What it does**: Stores the relational mapping of documents (`documents` table), pages (`pages` table), and chunks (`chunks` table) for incremental RAG indexing.
* **Why it's used**: Replaces the memory-heavy JSON cache to prevent RAM exhaustion when handling large corpora. It serves as the single source of truth for incremental index builds and document reconciliation.

### 2. Vector Index (`data/chroma_db/`)
* **What it is**: A local persistent folder generated by ChromaDB.
* **What it does**: Stores the HNSW (Hierarchical Navigable Small World) index graphs and the float embeddings (384-dimensions) computed by our local `all-MiniLM-L6-v2` model.
* **Why it's used**: Handles high-speed semantic nearest-neighbor queries (cosine similarity) on user inputs in under ~50ms.

### 3. Lexical Index (`data/bm25_index.pkl`)
* **What it is**: A binary pickle file containing the serialized `BM25Okapi` index from the `rank-bm25` library.
* **What it does**: Stores a word-to-document mapping of all text chunks along with a lookup list of the chunk dictionaries.
* **Why it's used**: Serializing the lexical index to disk allows the chatbot to load it instantly (~50ms) on startup. It performs exact keyword matching to complement the vector search.

### 4. Query Monitoring Database (`data/monitoring.db`)
* **What it is**: A local SQLite database containing the `query_logs` table.
* **What it does**: Persistently logs every conversation and query metric:
  * **Factual metrics**: Query string, generated response, and the exact chunks used as sources (saved as JSON text).
  * **Safety metrics**: Confidence score (%), confidence level (High/Medium/Low), and fact check verification status (`FULLY_SUPPORTED`, `PARTIALLY_SUPPORTED`, `UNSUPPORTED`, `API_ERROR`).
  * **Performance metrics**: Timestamps and execution latencies in milliseconds for each stage (Retrieval, Reranking, Generation/Validation, and Total Latency).
* **Why it's used**: Serves as the database source for the Streamlit **Monitoring Dashboard** tab, allowing real-time visualization of P95 latency, average confidence, and hallucination rates.
